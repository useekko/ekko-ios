# Ekko on iOS

Everything under `ios/`. Three shipping surfaces plus one shared core:

| Target | What it is | Why it exists |
|---|---|---|
| `Ekko` | SwiftUI app | Identity, the 24-word backup, contacts, safety numbers, settings |
| `EkkoKeyboard` | Custom keyboard extension | **The overlay.** iOS gives no other way to reach inside Instagram's or WhatsApp's native app |
| `EkkoSafari` | Safari web extension | The Chrome extension, running in Safari on iPhone (and macOS) |
| `EkkoCore` | Local Swift package | The protocol: crypto, wire format, chunking, BIP39, vault, broker |

## The one thing that must never break

**The same 24 words produce the same identity on iPhone and in the browser extension.** Restore
your phrase on a new phone and your fingerprint, your @handle, and everyone who can reach you are
unchanged. A message sealed in Chrome opens on iOS and vice versa.

This is not an aspiration, it is a test. `npm run ios:interop`:

1. generates wire vectors from the **real** `src/core/*.ts` (via esbuild, no mocks),
2. runs `swift test`, which asserts every one of them,
3. takes the handshake and message that Swift sealed and opens them with the TypeScript core.

If either implementation drifts, this fails. Run it after touching `src/core/` **or**
`ios/EkkoCore/`. The vectors live in `ios/EkkoCore/Tests/EkkoCoreTests/vectors.json` and are
regenerated by that script, so `swift test` alone still checks the TS -> Swift direction.

### Why it works with zero crypto dependencies

CryptoKit on the iOS 26 SDK ships **`MLKEM768` with a seed-based initializer that is byte-identical
to `@noble/post-quantum`** (both are FIPS 203, same `d‖z` seed expansion, verified). X25519, HKDF,
and SHA-256 are all CryptoKit too. The only primitive Apple does not provide is **XChaCha20-Poly1305**:
CryptoKit has the IETF 12-byte-nonce `ChaChaPoly`, and the wire format needs the 24-byte-nonce
extended variant. So `XChaCha.swift` implements HChaCha20 (the subkey derivation, ~40 lines) and
hands the rest to CryptoKit. It is checked against `@noble/ciphers` vectors.

> **Deployment target is iOS 26** purely because `MLKEM768` landed in CryptoKit there. If we ever
> need to go lower, add `apple/swift-crypto` and the type name does not even change.

## The keyboard is where the product lives

The native keyboard uses the messenger's real composer, matching the browser extension's familiar
editing model:

```
your fingers -> host composer -> tap Ekko's Seal -> EKK1M replaces the plaintext -> tap app Send
```

This is a deliberate usability/privacy trade: the native host app can observe the plaintext draft
until Seal is tapped. A system keyboard has only text-proxy insert/delete/cursor access and cannot
intercept the host app's Send button. Safari's web extension is different: its content script can
capture Instagram's DOM click/Enter event, prevent the send, then replace the composer first.

The keyboard keeps Return as Return. One 42pt state-aware bar shows the current recipient, an
iOS-owned Paste control, and one prominent Seal action; it temporarily becomes the contact picker,
sealed queue, or a real status/error when needed. There is no permanent instruction rail, second
Seal key, or duplicate private-draft panel. The complete compose surface is 270pt high and uses
the native iOS keyboard's neutral light/dark hierarchy, with coral reserved for Ekko actions.
Tapping Seal encrypts first and only then replaces the composer, so a setup/encryption failure
leaves the plaintext draft intact for editing. The QWERTY plane uses the full available width
without crowding or clipping the outer keys. Its letter, number, and symbol geometry, cap colors,
corner profile, expanded touch targets, Shift behavior, and held-Delete cadence are maintained in
`Shared/NativeKeyPlane.swift`; the shipping extension and KeyboardLab compile that same file.
The bottom row also opens an in-extension Emoji surface with nine categories, on-device recents,
tap-and-hold Delete, and an ABC return key. This avoids depending on the user's keyboard order—the
only public system-keyboard API advances to the next input mode and cannot select Emoji directly.

### Full Access: required, and here is the honest reason

iOS refuses a keyboard extension access to its own App Group container unless "Allow Full Access" is
on. The vault (your keys) lives in that container. So the toggle is not optional.

**The Ekko keyboard makes no network requests at all.** Directory lookups happen in the app, never
in the keyboard. Say this plainly to users; it is the true answer to a scary-looking switch, and it
stays true only as long as nobody adds a URLSession to `EkkoKeyboard/`. Do not.

### The send queue

The keyboard never puts a handshake in the host app. Account connections stage the post-quantum
setup in Supabase's `session_setups` table while the request is pending; RLS reveals it only after
acceptance, and the app derives the session before either person opens a messenger. Off-grid users
exchange the same setup as the one-time return invitation in Ekko itself.

A normal short message is therefore one `EKK1M:` token and one host-app Send after manual Seal,
including the first message. The queue remains only for genuinely long encrypted plaintext: when
the host clears one message chunk, `textDidChange` inserts the next. If setup has not happened,
sealing fails closed, preserves the draft, and tells the user to open Ekko; it never inserts a
four-part preamble.

### Reading messages

Safari uses the Ekko web extension to decrypt supported web messengers in place. Native apps do
not expose their incoming-message UI to a keyboard, so reading there is one explicit copy action:
long-press the sealed bubble, Copy, then tap the iOS-owned **Paste** control in the Ekko keyboard.
`UIPasteControl` transfers the copied string only after that tap, avoiding direct clipboard reads
and the recurring permission/nil behavior they cause on modern iOS. The keyboard opens the token
with the same `Engine.ingest` path as the iOS app (wire-compatible with the Safari extension).

Reading replaces the keys with an opaque, scrollable private reader; plaintext never enters the
messenger's composer. The reader shows sender/status context, actionable errors, multi-chunk
progress with **Paste next**, and **Reply** or **Done**. Closing it or leaving the keyboard clears
the displayed plaintext. Readers still accept legacy in-chat handshakes for compatibility, but
current iOS senders never create one.

This is deliberately not automatic: iOS gives a keyboard no API to inspect a received bubble, and
the system Paste control keeps clipboard access visible and user-directed.

## Storage

One file, `vault.json`, in the App Group container (`group.app.useekko`), shared by the app and the
keyboard. Written with `.completeFileProtection` (its key is wrapped by the device passcode and
evicted when the phone locks) and excluded from backups.

The browser vault is a scrypt-passphrase-encrypted blob because `chrome.storage.local` is
plaintext. iOS gives us Data Protection for free, so there is **no passphrase to type** — which is
also the only thing that makes the keyboard usable. Contacts and sessions do not need to port
between devices; the phrase re-derives the identity and a fresh handshake re-establishes sessions.

## Build, run, test

```bash
npm run ios:gen         # <- START HERE on a fresh clone. Builds the extension, syncs it into
                        #    the Safari target, and regenerates Ekko.xcodeproj.
open ios/Ekko.xcodeproj

npm run ios:interop     # the wire-compat proof (TS <-> Swift)
npm run ios:safari      # just the extension build + Safari sync
```

`ios/EkkoSafari/Resources/` is generated (a copy of `dist/`) and **gitignored**, like `dist/`
itself. A fresh clone has no Resources, so a build straight from Xcode fails with a missing-file
error until `npm run ios:gen` has run once. That is the one setup step.

`ios/project.yml` is the source of truth for the project. **Edit it, never the pbxproj.** The
`.xcodeproj` is committed as well so the project opens without installing xcodegen, but any change
must go through `project.yml` + `npm run ios:gen`.

Tests:

```bash
swift test --package-path ios/EkkoCore          # 24 core tests (crypto, wire, engine, interop)
cd ios && xcodebuild test -scheme Ekko \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
  -only-testing:EkkoKeyboardTests               # 20 keyboard tests (privacy, send queue, decrypt)

# end to end: the keyboard, inside Safari, sealing a real message
scripts/ios-sim-setup.sh                        # register the keyboard (as Settings would)
node scripts/ios-seed-sim.mjs                   # seed the App Group vault: identity + 1 contact
cd ios && xcodebuild test -scheme Ekko \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
  -only-testing:EkkoUITests/KeyboardInHostAppTests

# end to end: onboarding, from the welcome screen to the app, choosing off-grid
scripts/ios-reset-sim.sh                        # a simulator that has never seen Ekko
cd ios && xcodebuild test -scheme Ekko \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
  -only-testing:EkkoUITests/OnboardingFlowTests
```

### KeyboardLab: clean-room native parity

`ios/KeyboardLab` is a standalone Swift app and UI-test target, deliberately separate from
`Ekko.xcodeproj`. One `UITextView` switches between Apple's active English keyboard and the exact
`NativeKeyPlaneView` compiled into the shipping extension, under the same device, width, appearance,
and input state. It observes public rendered output only; it does not inspect or call Apple's private
keyboard implementation.

Use a clean iPhone Pro Max simulator whose keyboard list contains only English and Emoji, then run:

```bash
cd ios/KeyboardLab
xcodegen generate
xcodebuild build-for-testing -project KeyboardLab.xcodeproj -scheme KeyboardLab \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro Max' \
  -derivedDataPath /tmp/ekko-keyboard-lab-derived
xcodebuild test-without-building -project KeyboardLab.xcodeproj -scheme KeyboardLab \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro Max' \
  -derivedDataPath /tmp/ekko-keyboard-lab-derived
```

Each light/dark parity run captures Apple and Ekko on letters, `123`, and `#+=`. A pixel
connected-component analyzer gates visible cap geometry and exact surface/backing colors; public
accessibility frames independently gate the native-sized touch targets. A separate interaction test
drives one-shot Shift, double-tap Caps Lock, all plane transitions, typing, Space, tap Delete, and
held-Delete repeat. An Emoji test captures Apple's public Emoji input mode beside the shared Ekko
surface, then verifies category switching, insertion, recents, deletion, and the Emoji/ABC round
trip. Every run retains full screenshots, normalized crops, side-by-side comparisons, absolute
differences, and machine-readable JSON measurements.

`ios-reset-sim.sh` exists because **`simctl uninstall` is not enough**: the identity lives in the
App Group container, which is shared with the keyboard and therefore outlives the app. Skip the
reset and onboarding correctly resumes at the backup step, so the test finds no welcome screen. The
script clears both.

There is no simctl verb for "add a keyboard", so `ios-sim-setup.sh` writes the `AppleKeyboards`
preference the picker reads; the simulator needs a reboot afterwards for the keyboard daemon to
notice. `ios-seed-sim.mjs` writes the vault **using the TypeScript core**, so it doubles as a check
that Swift can load a vault another implementation wrote.

`EkkoUITests` is the one that proves the product rather than the parts. It launches Safari, switches
to Ekko with the globe key, checks the lock chip reads `Sealing to <contact>` (which can only happen
if the App Group entitlement worked), types five letters, **asserts Safari's text field is
byte-for-byte unchanged**, taps Seal, and only then finds an `EKK1` token in the field. It then puts
that real token on the simulator pasteboard, taps the keyboard's system Paste control, confirms the
private reader contains the original plaintext, and confirms the host field still contains only
the ciphertext. Verified green in the iOS 26 simulator.

## The Safari extension

`npm run ios:safari` copies `dist/` into `ios/EkkoSafari/Resources/` (generated; never hand-edit)
and makes two changes for Safari:

1. **The background service worker is rebuilt as an IIFE.** `esbuild.mjs` emits it as an ES module
   with code-splitting, which is right for Chrome. Safari **rejects `"type": "module"` on a service
   worker outright**, and the extension would load with no background at all: no keys, no decrypt,
   nothing. The sync script re-bundles `src/background.ts` as one self-contained classic script and
   drops the manifest key. The popup and onboarding stay ESM (page scripts, which Safari has
   supported for years).
2. **Host matches are widened** so `instagram.com` without the `www.` still gets a content script.

Verified in the simulator's actual Safari: `Uint8Array.toBase64` / `fromBase64` (the native TC39
base64 that `src/core/b64.ts` depends on, and which would silently break *all* crypto if missing)
are present, along with the other 7 APIs the core needs.

Users enable it in Settings > Apps > Safari > Extensions.

## Traps this project already fell into

- **`CODE_SIGNING_ALLOWED: NO` silently destroys the App Group.** This is the worst one, because
  nothing fails: with signing off, the ad-hoc signature is skipped, entitlements are never embedded,
  `containerURL(forSecurityApplicationGroupIdentifier:)` returns nil, the app quietly falls back to
  a private temp directory, and **the keyboard can never see the vault**. The app looks fine. The
  keyboard just always says "set up your identity". Keep it `YES` (with `CODE_SIGN_IDENTITY: "-"`
  and `CODE_SIGNING_REQUIRED: NO`, which still needs no team and no profile). Verify with
  `xcrun simctl get_app_container <sim> app.useekko.ios group.app.useekko` — it must print a path.
  Turning signing on then requires `GENERATE_INFOPLIST_FILE: YES` on the test targets.
- **Never add a folder reference named `Resources` to an iOS appex.** A `Resources/` directory
  inside the bundle makes CFBundle read it as a macOS-style bundle, and embedded-binary validation
  fails with the deeply unhelpful `error: Couldn't load Info dictionary`. Safari expects
  `manifest.json` at the **bundle root** anyway; the source folder is organizational only, and
  `project.yml` adds its files as flat resources with the real subdirectories (`icons`, `fonts`,
  `chunks`) as folder references. `ios-safari-sync.mjs` fails the build if esbuild starts emitting a
  subdirectory `project.yml` does not declare.
- **`.accessibilityIdentifier` on a SwiftUI container is inherited by every descendant.** An
  identifier on the keyboard's root view overwrote the identifier on all 26 letter keys, so every
  key announced the same id and none could be addressed. Put identifiers on leaves. (Labels do not
  behave this way; identifiers do.)
- **A custom keyboard's view is not matched by `XCUIApplication.keyboards`** — that query only ever
  finds the *system* keyboard. Once iOS remembers Ekko as the last keyboard used in an app, Ekko
  comes up first and `keyboards` matches nothing at all, which reads exactly like "the keyboard
  failed to load". Identify a custom keyboard by something it draws.
- **Do not set `PRODUCT_NAME: Ekko` on an extension target.** It renames the Swift module to `Ekko`
  too, which collides with the app's and produces `Multiple commands produce Ekko.swiftmodule`. The
  name users see in the keyboard picker is `CFBundleDisplayName`.
- **A hand-written `Info.plist` needs `CFBundleIdentifier`, `CFBundleExecutable`, and
  `CFBundlePackageType`.** Xcode only injects those when it generates the plist itself.
- **`.completeFileProtection` is iOS-only** and makes an atomic write fail with EPERM on macOS,
  where `swift test` runs. `Store.save` guards it with `#if os(iOS)`.
- Regex literals are not `Sendable`, so a `static let` regex fails Swift 6 strict concurrency. Make
  them computed `static var` (the literal is compiled at build time, so it costs nothing).

## Open in Safari (the launcher), and the home-screen wall

On iPhone, **in-place auto-decrypt happens only in Safari** — never in the native app (no overlay
API, no cross-app UI reading) and, crucially, **never in a "Add to Home Screen" web app**. A
home-screen web app runs in a standalone WebKit process with NO extension infrastructure, so the
content scripts, embeds and debug overlay simply are not injected there. This is why a user who adds
instagram.com to the home screen sees Ekko "disappear": same engine, different container.

The launcher is the answer: Settings -> **Open in Safari** lists Instagram / WhatsApp Web / Telegram
Web / Messenger, each opening the real URL via `UIApplication.open()`. That lands in Safari (a real
tab, extensions live), not standalone. `SafariMessenger.all` in `SettingsView.swift` holds the URLs;
they MUST stay on a page the manifest's content scripts match (see `scripts/ios-safari-sync.mjs`) or
the launcher opens a page Ekko never touches.

Caveat, stated in code: `UIApplication.open(https…)` goes to the DEFAULT browser. There is no public
API to force Safari specifically — but anyone who set up the Safari extension has Safari as default
(or the extension would be pointless), so this is correct in practice.

`EkkoUITests/SafariLauncherTests` pins the one thing that matters: tapping a row actually foregrounds
Safari (`com.apple.mobilesafari`), which a home-screen web app could never do.

## The account, and why it cannot hold your keys

Onboarding forks after the 24 words into the site's two modes: **"@you, everywhere"** (claim a
handle) or **"Off the grid"**. Both finish. The handle road passes through registration first —
Apple (native `SignInWithAppleButton`, id_token grant), Google, or the emailed 8-digit code — and
there is deliberately **no way to pick a handle without signing in**: the claim UI does not exist
until a session does. The account buys discovery and *nothing else* — take the ghost road, sign in
later or never, and the crypto is identical.

The reason it cannot be otherwise is the thing to keep hold of when someone asks "but how do the
private keys sync across devices?":

> **The 24 words ARE the sync.** The identity is *derived* from the phrase
> (`Recovery.deviceIdentity(phrase:)`), so the same phrase on an iPhone, an iPad and the Chrome
> extension yields the same keys, the same fingerprint, and the same contacts able to reach you.
> `npm run ios:interop` is the standing proof. There is nothing for a server to sync, and therefore
> nothing for a server to leak. An account is an entitlements holder, never a crypto identity
> (GOAL.md), and this is the design that makes that line cheap to hold rather than a sacrifice.

So the app has **two identities that must not be conflated**, exactly as `docs/ACCOUNTS.md` warns:

| | Where it lives | What it is |
|---|---|---|
| Identity | the 24 words → `vault.json`, on device | your keys. Cannot be recovered, reset, or signed out of |
| Account | Supabase, behind a session JWT | your @handle and your people. Signed out at will, replaceable |

`ios/Ekko/EkkoAccountClient.swift` is the client (zero dependencies: URLSession,
`ASWebAuthenticationSession`, Keychain). `ios/Ekko/AccountView.swift` is the UI: sign-in
(Apple + Google + emailed 8-digit code), handle claim, handle search, connection requests, and
linked socials. It hangs off the **Identity tab**, one tap away, rather than taking a tab of its own —
that tab is about the thing you cannot sign out of.

The claimed handle is mirrored into the vault's `username` field, which the Identity tab already
had a slot for and nothing was filling. It is a **display label**, not a key, and not the hashed
`platform_handles` mapping the standalone key directory keeps. Nothing bridges those two systems today.

An account connection also carries public post-quantum setup material through `session_setups`.
The server routes the same public handshake that an off-grid user trades as a return invitation; it
never receives either derived session key. The safety-number check remains the authentication step.

## Encrypted key backup (opt-in)

Answers "how do my keys get to my new phone?" without breaking the answer to "can Ekko read my
messages?" (no).

- **Format:** `src/core/backup.ts` (TypeScript) and `EkkoCore/Backup.swift` (Swift) — the same
  envelope, `{v, kdf, iter, salt, nonce, ct}`. XChaCha20-Poly1305 over the 24 words + contacts,
  under a PBKDF2-HMAC-SHA256 (600k) key.
- **`npm run ios:interop` proves both directions**: a blob sealed by the *browser extension* opens
  on the *phone*, and vice versa, contacts and timestamps intact. Without that, "back up here,
  restore there" is a guess.
- **PBKDF2, not the vault's scrypt**, because CryptoKit and swift-crypto ship no scrypt and porting
  it means hand-writing Salsa20/8. PBKDF2 is fast on a GPU, so the KDF is *not* what protects you —
  the **generated six-word passphrase** (~77 bits) is. A user-chosen passphrase sits behind an
  explicit toggle and is the only genuinely brute-forcible path. See `docs/THREAT_MODEL.md`.
- **The KDF parameters are AAD**, so a server handing back `iter: 1` fails the tag instead of
  yielding a key a laptop can grind. Tested on both sides.
- **Restore is on the welcome screen** ("Restore from my Ekko account"), because a new phone has no
  identity yet — offering it only after onboarding would be offering it too late.
- `Engine.restore(backup:passphrase:)` keeps each contact's original `addedAt` and `verified` flag.
  Routing them through `addContact` would stamp them all with today and silently drop the
  verification the user did by comparing safety numbers. Sessions are *not* in the blob: account
  sync or the off-grid return invitation re-establishes them before another message is sealed.

Server side (table, RLS, and three traps that cost real time): `docs/ACCOUNTS.md`.

## Not built yet

- **No directory integration in the app.** Claiming a *directory* @handle (the hashed, crypto-side
  one), looking peers up, and the `linkPlatform` / `handleStatus` flows all exist in the browser
  extension and in the server, but the iOS app does invite-based (paste / QR) contact exchange plus
  the *account*-side handle above. `EkkoCrypto.answerKeyChallenge` is ported and ready for it.
- **The browser extension cannot back up or restore yet.** `src/core/backup.ts` is built, tested and
  interop-proven, but the extension has no account sign-in (it would need Supabase auth via
  `chrome.identity.launchWebAuthFlow`), so nothing calls it there. Until that lands, the way to put
  the same identity in the extension and the app is the 24 words — which is exact, and works today.
- **The account path is not covered end to end by a test.** Registration is closed (invite-only), so
  `OnboardingFlowTests` drives the *off-grid* road all the way through and asserts the fork offers
  both mode cards, that the handle road leads to Apple + Google + email sign-in, and that no handle
  claim exists anywhere before a session does. Signing in needs an invited address;
  `scripts/account-smoke.mjs` covers the server side. **Google and Apple sign-in are live**
  (providers on — Apple verified against the live project 2026-07-16, client id = the app bundle
  id), and the buttons are wired, but **nobody has clicked either through on a device yet**: the
  `ASWebAuthenticationSession` round trip and the Apple sheet are the parts no simulator test here
  exercises. Sign in with Apple and the free team: a personal development team CANNOT carry the
  capability, so `Ekko/Ekko.entitlements` (Debug, the owner's-phone build) deliberately omits it
  and `Ekko/Ekko-store.entitlements` (Release, needs a paid team) adds it — see project.yml. The
  app checks its embedded provisioning profile at runtime and hides the Apple button in builds
  that could only throw; the simulator and store builds show it.
- **No autocorrect, no swipe typing, no dictation** in the keyboard. Those are enormous, and a
  keyboard you switch to for one sealed message does not need them. The lock toggle means Ekko never
  has to be the user's only keyboard.
- **The keyboard cannot know which app is hosting it** (iOS exposes no host bundle id), so unusually
  long encrypted messages still chunk at Instagram's conservative 900 characters everywhere.
- Not run on a physical device yet: everything above is simulator-verified.
